# 画像状態の伝票を自動で転記するシステム 実装手順

## 1. 目的
- 伝票PDFから項目を抽出し、確認・修正後にExcelを新規作成して出力するアプリケーションを構築する。
- フォーマットごとの枠座標・タグ情報はJSONで管理する。

## 2. スコープ
- 対応機能
  - モード選択画面
  - フォーマット登録機能
  - 文字認識機能
  - 抽出結果の確認・手動修正機能
  - Excel新規出力機能
- 保留事項
  - 基準点記録ロジック（フォーマット判別キー）
  - 重複チェック方法
  - フォーマット判別アルゴリズム

## 3. 推奨技術スタック
- バックエンド: Python + FastAPI
- フロントエンド: Streamlit または React + FastAPI
  - まずは開発速度重視でStreamlit推奨
- PDF処理
  - テキスト抽出: pdfplumber
  - 画像変換: pdf2image
  - 枠線検出: OpenCV
- OCR
  - pytesseract（ローカル）またはクラウドOCR（必要に応じて差し替え）
- Excel出力
  - openpyxl
- データ保存
  - `formats.json`（フォーマット定義）
  - `sessions/` 配下に抽出結果の一時保持（任意）

## 6. 実装ステップ

### Step 1. プロジェクト初期化
1. `narita` 配下にプロジェクトフォルダ作成。
2. `requirements.txt` を作成し主要ライブラリを追加。
3. `data/formats.json` を初期化（`{"formats": []}`）。
4. 最小UIとして「モード選択画面」だけ先に表示できる状態にする。

### Step 2. 共通基盤の実装
1. PDFアップロード処理を共通化。
2. PDFを画像化する処理を実装（ページ単位）。
3. 枠線検出サービスを作成。
   - 二値化
   - 輪郭抽出
   - 矩形候補フィルタ（最小サイズ、縦横比）
4. 座標系の統一
   - 画像座標とPDF座標の変換関数を用意。

### Step 3. フォーマット登録機能（A）
1. 空フォーマットPDFのアップロードUIを実装。
2. 枠線認識を実行し、検出結果を画像上でハイライト表示。
3. ユーザーが枠を選択し、タグ名を入力できるUIを実装。
4. 保存前バリデーション
   - タグ名の重複禁止
   - 未入力タグの禁止
5. `formats.json` へ保存。
6. 重複チェック（暫定）
   - 同一ページサイズ + フィールド数 + 主要枠位置の近似一致で判定。
   - 一致時はエラー表示して初期画面へ。

### Step 4. 文字認識機能（B）
1. 記入済みPDFのアップロードUIを実装。
2. 枠線認識を実行し、登録済みフォーマットと照合。
3. フォーマット未一致時はエラー表示して初期画面へ。
4. PDF種別判別処理を実装。
   - `pdfplumber` で文字オブジェクトが十分に取得できるかで判定。
5. 抽出処理の分岐
   - テキストベース: 登録座標から直接テキスト抽出。
   - 画像ベース: 座標でクロップしてOCR実行。
6. タグ名と抽出結果を一覧表示。

### Step 5. 手動修正UI
1. 抽出結果テーブルを編集不可で表示。
2. 「編集」ボタン押下でテキストボックス編集可能化。
3. 「保存」ボタンで修正値を状態に反映。
4. 複数PDFを続けて処理できるよう、結果をメモリまたは一時ファイルへ蓄積。

### Step 6. Excel新規出力
1. 「Excelダウンロード」ボタンを実装。
2. 蓄積データをタグ名ベースで列に整形。
3. `openpyxl` で新規Workbookを作成。
4. 1行目をヘッダー（タグ名）、2行目以降にデータを書き込み。
5. タイムスタンプ付きファイル名で `outputs/` に保存し、ダウンロード提供。

### Step 7. エラーハンドリング
1. 想定エラーと表示文言を統一。
   - PDF読み込み失敗
   - 枠線未検出
   - フォーマット未登録
   - OCR失敗
   - JSON保存失敗
2. 失敗時は初期画面へ戻れる導線を用意。

### Step 8. テスト
1. 単体テスト
   - 枠線検出
   - フォーマット照合
   - テキスト抽出
   - OCR抽出
   - Excel出力
2. 結合テスト
   - A機能の登録フロー
   - B機能の抽出からExcel出力まで
3. 異常系テスト
   - 未登録フォーマット
   - 破損PDF
   - OCRで空文字

### Step 9. 運用準備
1. ログ出力（処理時間、抽出件数、エラー内容）。
2. `formats.json` のバックアップ戦略を決定。
3. 将来拡張ポイントを定義。
   - 重複判定の高度化（画像特徴量）
   - フォーマット判別精度向上
   - OCRエンジン差し替え

## 7. まず作る最小実装（MVP）
- 対象を1ページPDF限定。
- フォーマット判別は暫定ロジック。
- OCRは日本語対応の基本設定のみ。
- UIは機能優先でシンプル構成。

MVP完了条件:
1. フォーマット登録ができる。
2. 記入済みPDFから抽出できる。
3. 抽出結果を画面で修正できる。
4. 新規Excelファイルを出力できる。

## 8. 保留事項の具体化タスク
1. 基準点記録仕様を確定。
   - 例: 左上から最初の特定サイズ枠をアンカーにする。
2. 重複チェック仕様を確定。
   - 例: IoU閾値 + フィールドタグ一致率。
3. フォーマット判別仕様を確定。
   - 例: アンカー一致 + 複数代表枠の相対位置一致。

### 10.5 現在のフォーマット判別方法（2026-04-17更新）
- 判別対象
  - 登録済みフォーマットごとに、`fields[].box_norm`（正規化座標）と `reference.anchor_box_norm` を参照する。
  - 認識対象PDFは1ページ目を画像化し、枠線検出結果を正規化座標へ変換して比較する。
- スコア計算
  - 各フォーマットに対して以下の総合スコアを計算し、最小スコアの候補を採用する。
  - `score = anchor_distance * 0.45 + layout_distance * 0.45 + aspect_distance * 0.10`
  - `anchor_distance`: フォーマットの期待アンカーに対し、検出枠のうち最も近い枠との距離。
  - `layout_distance`: 期待フィールドごとに最近傍検出枠との距離を平均した値（中心座標差 + サイズ差）。
  - `aspect_distance`: 登録フォーマットのページ縦横比と、認識画像の縦横比の差。
- 補正・ペナルティ
  - 枠数差ペナルティを `layout_distance` に加算する。
  - ペナルティは片方向ではなく、期待枠数と検出枠数の差分率を双方向で反映する。
- 判定条件
  - UIで指定したしきい値（デフォルト 0.18）以下なら自動一致とみなす。
  - しきい値を超えた場合は不一致扱いにして、手動フォーマット指定で続行可能。
- デバッグ表示
  - 不一致時に上位候補の内訳（アンカー距離、レイアウト距離、縦横比距離）を表示する。
  - これにより、誤判別が起きた際にどの要素が支配的かを確認できる。
